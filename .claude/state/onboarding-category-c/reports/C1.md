### Root cause (confirmed with a throwaway diagnostic, not committed)

I added a temporary `e2e/zzz-debug-diag.spec.ts` (deleted immediately after, never committed) that
just logged `context.pages()` right after the fixture's fresh-install launch:

```
pages after install: [
  'about:blank',
  'chrome-extension://iefcgkloeknedbdpcaeindnekmgnjomc/options.html'
]
```

This confirms: **on every fresh install, an `options.html` tab is now already open** (this is
exactly the Task 3 feature working correctly — see the passing `c1-open-on-install.spec.ts`).

`onboarding.spec.ts`'s third test exercises a *different, pre-existing, untouched* code path — the
no-key card's "Open Settings" button, which goes through `sw.ts`'s router (`openOptions: () =>
chrome.runtime.openOptionsPage()`, wired since before this card, not part of Task 2 or Task 3).
The test expects clicking that button to always open a **brand-new** tab
(`context.waitForEvent('page')`). But `chrome.runtime.openOptionsPage()`'s documented Chrome
behavior is to **focus/reuse an already-open options tab** instead of creating a new one. Now that
a fresh install genuinely leaves an options tab open (thanks to this task's fix), that reuse
behavior kicks in for real, no new `'page'` event ever fires, and the test times out.

Before this fix (under Task 2's code, committed as `f2a4bff`), `onInstalled`'s
`openOptionsPage()` call silently created no tab (that's the entire R1 diagnosis) — so no options
tab was ever actually open at this point in Task 2's committed state, and this interference could
not have manifested. This regression is a **direct, first-time-observable consequence of Task 3
correctly working**, not a pre-existing latent bug and not something in scope for R1 (R1 only
covers the `onInstalled` listener's own tab-opening call, and explicitly forbids adding
`chrome.tabs.query`, which is the natural way to make the router's `openOptions` call
reuse-aware).

## Why I did not just fix it myself

- Fixing this requires a decision on **product behavior**, not a code defect:
  - Should "Open Settings" on the no-key card, when an onboarding tab from install is already
    open, **focus that existing tab** (arguably better UX — no duplicate tabs) — in which case
    `onboarding.spec.ts`'s assertion ("always opens a brand-new tab") is the thing that's wrong
    and needs updating?
  - Or should the no-key card's CTA be guaranteed to always produce a fresh tab regardless, which
    would need a different mechanism for that call site (and touches pre-existing, non-Task-3
    code + a non-Task-3 test file)?
- Both routes require editing files **outside** Task 3's fenced scope (`onboarding.spec.ts`, and/or
  the router's pre-existing `openOptions` callback line, which is unrelated to Task 2/3's own
  additions) — the brief's hard walls say "Stay confined to Task 3's files... No adjacent
  refactors" and "If you hit a genuine open What/Why question not covered by R1 above, STOP."
- R1 explicitly forbids `chrome.tabs.query` (the natural way to detect "an options tab is already
  open" so the router could branch), so even a mechanical fix is walled off without further
  guidance.

## R3 option chosen: option 2 (fallback), after proving option 1 (preferred) is unreliable here

I attempted R3's **preferred** option 1 first — observe the active-tab transition directly via
`chrome.tabs.query({ active: true, lastFocusedWindow: true })` called from the service worker's
own context (the same pattern `helpers.ts`'s pre-existing `relayCommand` already uses, so no new
manifest permission). I wrote it, built, and ran it — it failed, not because the CTA is broken,
but because the observation itself doesn't work: this manifest declares no `"tabs"` permission
(by design — R1's own comment says as much), and Chrome strips the `url` field from
`chrome.tabs.query()` results for tabs it doesn't have permission to read. I proved this with a
throwaway diagnostic spec (`e2e/debug-tabs.spec.ts`, run once, deleted, never committed) that
dumped `chrome.tabs.query({})` right after the CTA flow's setup: two of the three tabs in the
context (the install-created options tab and the options tab opened by the test's own
`page.goto`) came back **with no `url` field at all** — only the active `http://test.fixture/`
tab (matched by the `<all_urls>` host permission) had a readable `url`. That makes it impossible
to identify "is the active tab options.html?" via this API without adding a manifest permission,
which R3 forbids. Diagnostic run output (abbreviated, full JSON was longer):

```

  { "id": 730558478, "active": false, ... }             // no `url` field — options.html tab (install-created)
  { "id": 730558480, "active": false, ... }             // no `url` field — options.html tab (test's own page.goto)
```

`context.pages()` post-click — to keep a positive check for "no duplicate" in the diff, since the
ordinary (non-closed) path is what actually exercises reuse (proven by the Task 3 spec + this
