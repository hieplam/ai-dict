## rules → project rules doc

- **Target:** `.c3/rules/` + `.c3/refs/` (this repo's canonical conventions live in the C3 model, not a `docs/rules.md`)
- **Current state:** exists — 5 enforced rules (`rule-api-key-isolation`, `rule-domain-purity`, `rule-gate-runtime-messages`, `rule-sanitize-model-output`, `rule-typed-errors`) plus refs that explain the "why" (`ref-core-dependency-rule`, `ref-wire-protocol-validation`, `ref-dependency-injection`, `ref-kv-storage-prefixes`, `ref-web-components-shadow-dom`). Project conventions also live in `CLAUDE.md`.
- **Inputs:** `concerns.rules.obey[]`, `concerns.rules.avoid[]`
- **Steps:**
  1. Rules are authored through the **C3 CLI** (`.c3/` is CLI-only — never hand-edit). Each `avoid[]` item maps to a NEVER-style constraint; each `obey[]` item to an enforced rule or ref.
  2. The lean dependency rule (one-way deps, dependency-free `packages/app/src/domain/`, communication only via `packages/app/src/ports.ts`) is enforced in code by `scripts/check-dep-direction.mjs` (run in the `lint` concern).
  3. Security rules S1/S3/S4 (api-key-isolation, gate-runtime-messages, sanitize-model-output) and `typed-errors` / `domain-purity` are the ones a reviewer/agent must check on every change.
- **Verify:** `c3 check` validates the C3 docs; `c3 lookup <file>` resolves which component + rules own a file. Every `avoid[]` item has a corresponding rule/ref or a CLAUDE.md directive.

## lint → linter config

- **Target:** `eslint.config.mjs` (flat config, `concerns.lint.config`)
- **Current state:** exists — flat ESLint config at repo root, plus the custom `scripts/check-dep-direction.mjs` dependency-direction guard, Prettier for formatting, and `knip` for dead-code/unused-dep detection.
- **Inputs:** `project.language`, `project.runtime`, `concerns.lint.tool`, `concerns.lint.strict`, `concerns.lint.formatter`, `concerns.lint.extra`
- **Steps:**
  1. Lint config is `eslint.config.mjs`; the lint entry point is `bun run lint` → `bun scripts/check-dep-direction.mjs && eslint .` (dep-direction check runs first, then ESLint).
  2. Formatting is Prettier, invoked as `bun run format` (write) / `bun run format:check` (verify).
  3. `knip` runs via `bunx knip` to flag dead exports / unused deps.
- **Verify:** on a fresh checkout, `bun install --frozen-lockfile && bun run lint && bun run format:check` runs clean.

## CI lint → CI workflow

- **Target:** `.github/workflows/ci.yml` (`concerns.ci.workflow`)
- **Current state:** exists — GitHub Actions, triggered on `push: [master]`, `pull_request`, and a nightly cron (which makes `dep-audit` + `gitleaks` blocking). All jobs install Bun via `.bun-version` and cache `~/.bun/install/cache`.
- **Inputs:** `project.host`, `project.runtime`, `concerns.ci.provider`, `concerns.ci.runs[]`, `concerns.ci.triggers`
- **Steps:**
  1. Workflow is GitHub Actions at `.github/workflows/ci.yml`. Base job `install` runs `bun install --frozen-lockfile`; downstream jobs `needs: install`.
  2. Each entry in `concerns.ci.runs[]` is its own job: `typecheck`, `lint`, `format-check`, `test-unit`, `test-component`, `test-contract`, `build-chrome`, `build-safari`, `e2e-chrome` (Playwright + unpacked extension under `xvfb`, uploads traces on failure), `coverage-gate` (per-package vitest thresholds), `sonarcloud` (quality-gate.wait), `secret-scan` (gitleaks), `knip`, `dep-audit`.
  3. The lint gate runs the SAME command the `lint` concern produced: `bun run lint`. The format gate runs `bun run format:check`.
- **Verify:** the workflow is valid YAML and its `lint` / `format-check` jobs run the same commands as the local `lint` concern. Release automation lives in the separate `release-please.yml` / `release.yml` workflows.

## context → C3 architecture model (no separate wiki)

- **Target:** `.c3/` (the project's queryable C3 model) + `CLAUDE.md` — `concerns.wiki.enabled: false`, so **no `wiki/` tree and no block-wiki-edits hook are scaffolded.**
- **Current state:** exists — `.c3/` holds the component docs (`c3-1` app, `c3-2` extension-chrome, `c3-3` extension-safari), `refs/`, `rules/`, `code-map.yaml`, ADRs, and `c3.db`. This IS the context/knowledge system; `CLAUDE.md` summarizes the shape and points at it.
- **Inputs:** `concerns.wiki.enabled` (false → do nothing), `concerns.wiki.context_system` (`c3`)
- **Steps:**
  1. Do **not** run the Ymir wiki scaffold. The context concern is satisfied by C3.
  2. To consult context: use the `c3` CLI (`c3 lookup <file-or-glob>`, `c3 list`, `c3 graph <id> --format mermaid`, `c3 check`) or read the `.c3/` model directly.
  3. Never hand-edit `.c3/` — it is CLI-only.
- **Verify:** `c3 check` reports the model is valid (or read `.c3/code-map.yaml` + component dirs directly if the CLI is unavailable).
