# Escalation: C1

**Reason:** needs_direction

## Context
The Hunter's fix (R1's `chrome.tabs.create` swap) works correctly and the new e2e test is green — but it exposed a genuine, deterministic regression in the pre-existing `onboarding.spec.ts` suite that R1 doesn't cover and that sits outside C1's declared file scope. I traced the root cause myself to confirm the Hunter's diagnosis is correct before deciding whether this is mine to resolve.

**What's happening:** `onboarding.spec.ts`'s third test navigates a page to `options.html`, then away to a fixture page, then clicks "Open Settings" and expects a *new* `context.waitForEvent('page')`. Before C1's fix, `onInstalled`'s `openOptionsPage()` call silently created no tab (that's the whole R1 bug), so no options tab existed at click time — a fresh tab opened, test passed. Now that `chrome.tabs.create` genuinely leaves an options.html tab open from install, `chrome.runtime.openOptionsPage()`'s real Chrome behavior (focus/reuse an already-open same-URL tab rather than create a new one) kicks in for the first time, no new `'page'` event ever fires, and the test times out deterministically.

The plan's own §5 explicitly asserted "no existing test's assertions change" — that assumption is now proven false, and it's false specifically because the assumption was built on a mechanism (`openOptionsPage()`) that, per R1, never actually worked. Fixing this means either (a) changing `onboarding.spec.ts`'s assertion to accept focus-of-existing-tab as correct, or (b) giving the no-key card's CTA its own tab-creation mechanism — both touch files outside C1's declared scope (`onboarding.spec.ts`, or the pre-existing router `openOptions` callback), and neither is covered by R1 or R2. This isn't in my Warchief "How" authority to quietly decide — it's exactly the kind of plan-vs-reality mismatch the campaign's own standing rule (R2.3) reserves for escalation rather than unilateral redesign.

State is preserved: branch `feature/C1OpenOnInstall` and worktree `.claude/worktrees/c1-open-on-install` are intact, with the R1 mechanism swap and new e2e spec sitting uncommitted (deliberately, since Task 3 isn't resolved), full evidence in `.claude/state/onboarding-category-c/reports/C1.md`.

NEEDS_DIRECTION: Now that C1's fix causes a genuine options.html tab to stay open from install, `chrome.runtime.openOptionsPage()`'s built-in reuse behavior makes it focus that tab instead of opening a new one — which breaks `onboarding.spec.ts`'s "no-key card... Open Settings opens the options page" test (it waits for a `'page'` event that never fires; deterministic, reproduced twice). Should that test's assertion be updated to accept "an existing options.html tab is focused/visible" as the correct outcome (my read: this is better UX, no duplicate tabs, and no product promise changes), or does the no-key card's CTA need a distinct tab-creation mechanism of its own? Either fix touches files outside C1's declared scope (`onboarding.spec.ts` and/or the pre-existing router `openOptions` callback), so I'm not deciding it unilaterally.
